## 一、产品定位
- **是什么**：个人一站式「工程账本」——记录各工程的超管/普通用户链接、账号密码、简介，也含日常网页/论坛/代家人等密码。
- **核心原则**：**所有条目平等**，无独立的"密码本"概念，靠**用户自定义分类**组织。
- **多端**：浏览器 / 安卓手机壳，同一份数据（COS 密文）。
- **零知识**：明文只在浏览器内存，服务器（SCF/COS）永远只见到密文。
- **不是什么**：不是团队协作工具、不是共享密码库（分享是"只读单条、限时"），不是密码强度审计器。

## 二、核心架构决策（已定稿）
| 决策 | 内容 | 为什么 |
|---|---|---|
| 前端与后端分离 | SCF 只做**纯 API**（返回 JSON，绝不返 HTML）；前端 `index.html`+`static/` 托管 GitHub Pages / CloudStudio | SCF Web 函数若直返 HTML，浏览器会当**附件下载**而非渲染（已踩坑） |
| 零知识 | 浏览器内 PBKDF2/AES-GCM 加解密；SCF 永不见明文 | 服务器被攻破也拿不到密码 |
| 自算密钥工具 | `SHA-256(UTF-8(明文))`→64位小写 hex | 对齐 word-dictation / 匿名互评，方便把密钥贴进环境变量 |
| 运行时 | Python 3.10+ / Flask 3.x；依赖打包 vendor；Web 函数**不设 Handler** | 甩掉旧 3.6 的坑 |
| 部署链路 | GitHub Actions：`deploy-scf`→SCF + `deploy-pages`→Pages；凭证仓库 Secrets `TENCENT_SECRET_ID/KEY` | 推 main 即自动部署 |
| 跨域 | 前端经浏览器内 `API_BASE`（localStorage，已预填你的 SCF 地址）调 SCF，SCF 加 CORS 头 | 前后端不同源 |

## 三、数据安全模型（v3.1 信封加密）
> 🔲 已定稿待开发。设计目标：改密秒级、支持应急恢复、绝密区物理隔离。

- **双区隔离**
  - 普通区 `vault.json`（主密码派生 `vaultKey` 加密）
  - 绝密区 `secret.json`（独立 `secretKey` 加密，纯浏览器、不进 env，与普通区物理隔离）
- **信封加密**：每区一把随机数据密钥 DEK 真正加密数据；DEK 被密码派生密钥"包裹"存 COS `auth.json`
  - `DEK_normal` 两把锁：主密码**或**应急恢复密码 R 都能解
  - `DEK_secret` 一把锁：仅绝密密码能解，无恢复包
  - 改主密码 = 只重包一次 DEK（数据不重加密，快）
- **登录凭据放 COS 而非 env**：env 只放永不变的（`SESSION_SECRET` + COS 凭据）；`ADMIN_PWD` env 仅首次引导，首登后权威转移到 COS `auth.json` → 重置只写 COS，永不碰 env
- **应急恢复（数字遗产）**：自定义恢复密码 R → 生成无限期公开 `/recover` 链接；R 由用户**手动拆分**给多人（未用 Shamir）。继承人拼齐 R → 解普通区 → 强设新主密码 → 看资产统计，**碰不到绝密区**。恢复端点限流防爆破
- **系统重置（factory wipe）**：登录后"重置整个系统" → 二次确认 + 手输 **`DELETE`**（区分大小写）→ 清空 COS 全部对象；执行前强制先导出加密备份
- **登录 IP 记录**：SCF 从 `X-Forwarded-For`/`X-Real-IP` 取真实 IP，成功时记入日志（建议加密存）

## 五、移动端壳（Android）
- **动机**：手机直接当 App 用，不必开微信/笔记本复制链接
- **A 方案 PWA（✅ 原型已上线）**：`manifest.webmanifest` + `sw.js` + 蓝底白锁图标；手机 Chrome「添加到主屏幕」即全屏 App，无需 APK/签名
  - 名称「工程账本」/ 简称「账本」；主题色 `#4f6ef7`；竖屏 standalone
- **B 方案 TWA APK（⏸ 待确认）**：Bubblewrap 包成 `.apk`，需自签 keystore——能用 Android Keystore 把密钥**绑死硬件**（解决 PWA 落盘风险）
- **登录/会话模型（🔲 待开发，并入 v3.1）**：
  - **长窗口免密**：首次输主密码后缓存 `vaultKey`（localStorage/IndexedDB），**解锁窗口 5–7 天**；窗口内打开即解锁、不弹密码；关页/切后台不锁
  - 仅两种重输触发：①窗口到期 ②手动「立即锁定」
  - **双令牌每日轮换**：refresh token（5–7 天，换发 access token）+ access token（≈1 天，真正授权 API）；每日用 refresh 换新，防 cookie 泄露
  - **硬伤**：纯 PWA 无法硬件绑定，缓存有被其他 App 读到的理论风险（root 设备）；缓解 = 「立即锁定」+ 窗口到期。B 方案才能根治
- **构建方式确认（⏸ 开工前必问）**：要 A / B / 还是 A+B？
